基于 Three.js、APlayer 和 Meting API 的独立音乐页面实现与效果设计

本文记录一个用于静态博客的独立音乐页面实现。页面部署在 GitHub Pages 中，通过 VuePress Theme Hope 的导航入口跳转访问。

1. 页面定位

整体结构分为两部分：

  • 背景层：Three.js 粒子树、扫描环、网格和漂浮粒子
  • 音乐层：APlayer JS API 负责播放器交互，Meting 接口负责从网易云音乐逐首拉取歌曲数据

实现目标是让页面作为一个独立的 listen.html 存在，不依赖 VuePress 路由，也不需要后端服务（风格仿照明日方舟终末地科技）。

2. 技术栈

页面主要使用以下技术：

HTML / CSS
Three.js
APlayer
Meting API
Web Audio API
GitHub Pages
VuePress Theme Hope

各部分职责如下：

模块作用
Three.js绘制粒子树、扫描环、网格和环境粒子
APlayer提供播放器核心能力
Meting API根据歌曲 ID 获取音乐链接、封面、作者、歌词等数据
Web Audio API读取音频频谱，驱动背景轻微响应
VuePress提供博客导航入口
GitHub Pages托管最终静态页面

3. 页面结构

页面主体结构比较简单：

<canvas id="bg"></canvas>

<div class="vignette"></div>
<div class="scanline"></div>

<div class="header">
  <div class="header-sub">MACRO FIELD</div>
  <div class="header-main">LISTEN</div>
  <div class="header-tag">SIGNAL / BREATH / MEMORY</div>
</div>

<div id="aplayer"></div>

其中：

  • canvas#bg 用于 Three.js 场景
  • .vignette 和 .scanline 用于视觉叠层
  • .header 是页面文字信息
  • #aplayer 是 APlayer 的挂载容器

音乐播放器没有手写完整的音频控制逻辑，而是交给 APlayer。页面只负责歌曲数据获取、样式定制和视觉联动。

4. Three.js 背景层

背景主体是一个抽象粒子树。它不是导入模型，而是用递归分枝生成点云和线段。

场景结构大致如下：

scene
├── world
│   ├── baseMesh
│   ├── branchLines
│   ├── twigLines
│   ├── branchMesh
│   ├── crownMesh
│   ├── coreMesh
│   └── ringGroup
├── gridGroup
└── ambientParticles

树本体放在 world 组中，方便统一控制旋转、缩放和音频响应。

const world = new THREE.Group();
world.position.set(0, -0.12, 0.36);
scene.add(world);

相机初始位置经过调整，避免树过大：

const camera = new THREE.PerspectiveCamera(
  46,
  window.innerWidth / window.innerHeight,
  0.1,
  180
);

camera.position.set(0, 2.12, 10.15);

5. 粒子树生成与交互动画设计

树的生成使用递归函数。每一段分枝会生成线段骨架、枝干点云和末端树冠粒子。

简化后的逻辑如下：

function grow(ox, oy, oz, dx, dy, dz, len, depth, spread) {
  if (depth <= 0 || len < 0.07) return;

  const steps = Math.max(8, Math.floor(len * 22));

  for (let i = 1; i <= steps; i++) {
    const t = i / steps;

    const x = ox + dx * len * t;
    const y = oy + dy * len * t;
    const z = oz + dz * len * t;

    addPoint(branchPts, branchCols, x, y, z);
  }

  const childCount = depth > 5 ? 4 : depth > 3 ? 3 : 2;

  for (let i = 0; i < childCount; i++) {
    const nextDir = computeBranchDirection(dx, dy, dz, spread);
    grow(x, y, z, nextDir.x, nextDir.y, nextDir.z, len * 0.6, depth - 1, spread);
  }
}

为了让树在画面中稳定可见，点云材质使用不透明渲染：

const material = new THREE.PointsMaterial({
  size,
  vertexColors: true,
  sizeAttenuation: true,
  transparent: false,
  depthTest: true,
  depthWrite: true,
  blending: THREE.NoBlending
});

这样可以减少透明排序问题。尤其是在画面中同时存在树、环、网格和环境粒子时，不透明点云的层级关系更稳定。

视觉层级

页面的层级设计遵循一个原则：主体，背景，动态。

整个画面可以分成三层：

前景：粒子树
中景：扫描环
背景：网格、暗角、漂浮粒子

粒子树是画面的主要视觉对象，放在中心位置。它由主干、分枝、树冠点云和核心粒子组成。为了保证形态清楚，树本体使用不透明点云和线段骨架，而不是大量透明叠加。这样即使背景里有扫描环和网格，树的轮廓也不会被冲掉。

扫描环处在中景位置，主要提供空间参照。环形结构会缓慢旋转，让画面有持续运动感，但它整体被放到树的后方，亮度也比树低。这样可以保留透视感，同时不影响树的识别度。

网格和漂浮粒子属于背景层。网格用来建立纵深关系，漂浮粒子提供轻微的环境运动。它们的运动幅度较小，颜色也更暗，避免和主体竞争。

音乐播放时，背景会随音量产生轻微变化。整体音量会影响树冠的呼吸幅度和扫描环的转速，低频会让核心粒子和树体产生轻微缩放，高频会影响漂浮粒子的运动幅度。变化幅度控制得比较小，主要用于增强音乐和画面之间的联系，而不是做强烈的频谱动画。

这种分层方式的重点是把视觉职责拆开：

树：负责主体识别
扫描环：负责空间参照
网格和粒子：负责环境氛围
音量响应：负责音乐联动

因此页面虽然有多个动态元素，但整体不会混乱。树保持中心地位，背景提供空间和运动，音量变化提供轻微反馈。

6. 音乐部分设计

音乐部分采用：

APlayer JS API + Meting API

没有直接使用网易云的外链，也没有手动维护每首歌的真实音频地址。页面只维护歌曲 ID，然后逐首请求 Meting 接口。

基本配置如下：

const SONG_IDS = ['3371305930', '2700421303', '65778'];

const SERVER = 'netease';

const API_BASE = 'https://api.injahow.cn/meting/';
const API_FALLBACK = 'https://api.i-meto.com/meting/api';

SONG_IDS 只保存网易云歌曲 ID。后续想换歌，只需要改这个数组。

7. 逐首拉取歌曲信息

Meting 接口请求方式如下：

async function fetchSong(id, useFallback = false) {
  const api = useFallback ? API_FALLBACK : API_BASE;

  const url =
    `${api}?server=${SERVER}&type=song&id=${id}&r=${Math.random()}`;

  const res = await fetch(url, {
    cache: 'no-store'
  });

  if (!res.ok) {
    throw new Error(`HTTP ${res.status}`);
  }

  const data = await res.json();

  if (!data || !data[0]) {
    throw new Error('empty response');
  }

  return data[0];
}

这里加了 r=${Math.random()}，主要是避免部分接口或浏览器缓存旧结果。

初始化时逐首请求：

async function loadSongs() {
  let rawSongs;

  try {
    rawSongs = await Promise.all(
      SONG_IDS.map((id) => fetchSong(id, false))
    );
  } catch (error) {
    rawSongs = await Promise.all(
      SONG_IDS.map((id) => fetchSong(id, true))
    );
  }

  return rawSongs.map((song, index) => ({
    name: song.name || `Track ${index + 1}`,
    artist: song.artist || 'Unknown Artist',
    url: song.url || '',
    cover: song.pic || '',
    lrc: song.lrc || ''
  }));
}

主接口失败时，会自动切换到备用接口。两个接口都失败时，页面可以显示错误状态，而不是让播放器空白。

8. 初始化 APlayer

拿到歌曲数据后，通过 APlayer JS API 初始化播放器。

async function initPlayer() {
  const audioList = await loadSongs();

  const ap = new APlayer({
    container: document.getElementById('aplayer'),
    audio: audioList,
    fixed: false,
    mini: false,
    autoplay: false,
    mutex: true,
    listFolded: false,
    listMaxHeight: '280px',
    lrcType: 3,
    order: 'list',
    loop: 'all',
    volume: 0.8,
    theme: '#7dd3c8'
  });

  window.__PLAYER__ = ap;
}

APlayer 负责以下功能：

功能说明
播放 / 暂停APlayer 内置按钮
上一首 / 下一首APlayer 内置列表控制
进度拖拽APlayer 内置进度条
音量控制APlayer 内置音量控件
列表展开 / 折叠listFolded 和列表按钮
随机 / 顺序通过 APlayer API 切换
单曲 / 列表循环通过 APlayer API 切换
歌词lrcType: 3，使用外部歌词文本

这里没有使用 <meting-js> 标签自动生成播放器，而是手动调用 APlayer API。这样做的好处是可以控制歌曲加载、错误处理、样式覆盖和背景联动。

9. APlayer 事件与页面联动

APlayer 初始化后，可以监听播放状态：

ap.on('play', () => {
  document.body.classList.add('is-playing');
});

ap.on('pause', () => {
  document.body.classList.remove('is-playing');
});

ap.on('ended', () => {
  console.log('track ended');
});

如果需要将当前歌曲信息同步到页面自定义 HUD，也可以读取：

const current = ap.list.audios[ap.list.index];

document.querySelector('#track-title').textContent = current.name;
document.querySelector('#track-artist').textContent = current.artist;

APlayer 的音频元素可以通过 ap.audio 访问，这一点对 Web Audio API 联动很有用。

10. Web Audio API 频谱联动

背景粒子树不直接依赖播放器 UI，而是通过 ap.audio 接入 Web Audio API。

const AudioContext = window.AudioContext || window.webkitAudioContext;
const ctx = new AudioContext();

const source = ctx.createMediaElementSource(ap.audio);
const analyser = ctx.createAnalyser();

analyser.fftSize = 128;
analyser.smoothingTimeConstant = 0.78;

source.connect(analyser);
analyser.connect(ctx.destination);

const data = new Uint8Array(analyser.frequencyBinCount);

每帧读取频谱数据：

function updateAudioLevel() {
  analyser.getByteFrequencyData(data);

  const bass = average(data, 0, 7);
  const mid = average(data, 8, 20);
  const treble = average(data, 21, data.length);

  window.__LISTEN_AUDIO__ = {
    level: average(data, 0, data.length),
    bass,
    mid,
    treble
  };

  requestAnimationFrame(updateAudioLevel);
}

Three.js 动画循环中读取这个对象：

const fx = window.__LISTEN_AUDIO__ || {
  level: 0,
  bass: 0,
  mid: 0,
  treble: 0
};

然后只做轻量响应：

crownMesh.scale.setScalar(1 + fx.level * 0.04);
coreMesh.scale.setScalar(1 + fx.bass * 0.02);
ringGroup.rotation.z += 0.001 + fx.treble * 0.0008;

这里没有让音乐强烈控制画面，而是保持背景稳定。音乐页的重点仍然是播放体验，不是频谱可视化演示。

11. 播放模式控制

如果需要自定义播放模式，可以通过 APlayer API 管理。

例如切换随机播放：

function setShuffle(ap, enabled) {
  ap.options.order = enabled ? 'random' : 'list';
}

切换循环模式：

function setLoop(ap, mode) {
  // mode: 'all' | 'one' | 'none'
  ap.options.loop = mode;
}

也可以在 UI 上做成几个模式按钮：

ORDER
SHUFFLE
REPEAT ONE
REPEAT ALL

点击按钮时只修改 APlayer 的配置，并更新按钮状态。

12. 样式定制

APlayer 默认样式偏通用，需要通过 CSS 覆盖，让它融入页面右下角面板。

容器可以这样处理：

#aplayer {
  position: fixed;
  right: 0;
  bottom: 0;
  width: 360px;
  z-index: 20;
}

面板背景和边框：

.aplayer {
  background: rgba(5, 10, 22, 0.92) !important;
  border-top: 1px solid rgba(125, 211, 200, 0.18);
  border-left: 1px solid rgba(125, 211, 200, 0.18);
  backdrop-filter: blur(24px);
  color: #dce8f2;
}

列表区域：

.aplayer-list {
  background: rgba(5, 10, 22, 0.96) !important;
  border-top: 1px solid rgba(125, 211, 200, 0.08);
}

.aplayer-list li {
  border-top: 1px solid rgba(255, 255, 255, 0.03) !important;
}

.aplayer-list li:hover {
  background: rgba(125, 211, 200, 0.05) !important;
}

主题色保持和页面一致：

theme: '#7dd3c8'

这样播放器不会像外部组件直接贴上去，而是和页面视觉统一。

13. 错误处理

Meting 接口是公共接口，稳定性不能完全保证。因此需要处理失败情况。

基本策略：

请求主接口
  ↓
失败后请求备用接口
  ↓
仍然失败则显示错误状态
  ↓
保留页面背景和本地上传入口

错误状态可以简单写成：

try {
  const audioList = await loadSongs();
  initAPlayer(audioList);
} catch (error) {
  console.error(error);
  showPlayerError('音乐接口暂时不可用');
}

不要让页面因为接口失败而整体崩溃。音乐只是页面的一个模块，背景和其他 UI 应该继续正常显示。

14. 本地文件补充

如果希望页面在接口不可用时仍能测试播放，可以保留本地文件上传。

<input type="file" id="file-input" accept="audio/*" multiple>

本地文件可以转成 Object URL：

const url = URL.createObjectURL(file);

ap.list.add([{
  name: file.name.replace(/\.[^/.]+$/, ''),
  artist: 'Local File',
  url,
  cover: '',
  lrc: ''
}]);

这部分不影响线上音乐源，只是方便调试和备用。

15. 与 VuePress / GitHub Pages 的关系

音乐页最终作为静态文件部署。

构建流程：

pnpm docs:build

确认输出文件存在：

ls src/.vuepress/dist/listen.html

然后将构建产物推送到 GitHub Pages 仓库。

如果使用 username.github.io 根域名仓库，base 一般保持 / 即可。

如果是项目页，例如：

https://username.github.io/project-name/

则需要在 VuePress 配置中设置：

base: "/project-name/"

当前音乐页是独立 HTML，部署后可以直接访问，不需要经过 VuePress 路由。

16. 实现总结

这个页面最终由两条线组成：

第一条是视觉线：

Three.js
  → 递归粒子树
  → 后置扫描环
  → 背景网格
  → 鼠标视差
  → 音频轻量响应

第二条是音乐线：

SONG_IDS
  → Meting API 逐首拉取
  → 标准化歌曲数据
  → APlayer JS API 初始化播放器
  → 播放状态同步到背景动效

这种结构的优点是模块边界比较清楚。

后续如果要替换音乐源，只需要改 SONG_IDS 或 SERVER。如果要替换视觉背景，也不会影响 APlayer 的播放逻辑。

17.threejs讨论

Three.js 的价值主要在于它把浏览器变成了一个轻量的实时 3D 渲染环境。对于静态博客来说，这一点比较有用：页面不再只能展示文字、图片和视频，也可以加入可交互的三维背景、模型展示、仿真结果可视化或数据驱动的空间图形。

后续如果想进一步提高画面完成度，可以把程序化生成的粒子树替换为自己的三维模型，或者把两者结合起来使用。例如：

Blender 建模
  → 导出 GLB / glTF
  → Three.js 加载模型
  → 添加粒子、光效和交互
  → 接入音乐音量响应

比较推荐的流程是使用 Blender 制作主体模型，然后导出为 .glb 文件。glb 是 glTF 的二进制格式，模型、材质、贴图和动画可以打包在一个文件里，适合网页加载。

在 Blender 中处理模型时，需要注意几个点：

1. 控制面数，避免模型过重
2. 应用缩放和旋转变换
3. 调整模型原点，方便 Three.js 中旋转和定位
4. 给对象和材质命名，便于代码中单独控制
5. 尽量合并贴图，减少网络请求
6. 导出为 GLB 格式

导出后可以放到 VuePress 的静态目录中：

src/
└── .vuepress/
    └── public/
        └── models/
            └── tree.glb

构建后访问路径通常是：

/models/tree.glb

Three.js 中可以用 GLTFLoader 加载：

import { GLTFLoader } from 'three/examples/jsm/loaders/GLTFLoader.js';

const loader = new GLTFLoader();

loader.load('/models/tree.glb', (gltf) => {
  const model = gltf.scene;

  model.position.set(0, -1.2, 0);
  model.scale.setScalar(1.0);

  scene.add(model);
});

如果模型中带有动画，也可以使用 AnimationMixer 播放：

const mixer = new THREE.AnimationMixer(model);

gltf.animations.forEach((clip) => {
  mixer.clipAction(clip).play();
});

然后在动画循环里更新：

mixer.update(deltaTime);

这种方式适合做更具体的主体物，比如树、机械结构、装置、建筑或角色剪影。程序化粒子则可以继续作为外层效果使用，例如模型表面的粒子流、背景漂浮点、扫描环或音频响应光效。

也就是说，后续可以形成两层结构：

Blender 模型：负责主体形态
Three.js 程序化效果：负责动态、交互和音乐响应

如果模型体积较大，还可以继续做优化：

Draco：压缩模型几何
KTX2：压缩贴图
LOD：根据距离切换不同精度模型
懒加载：进入页面后再加载模型

对于静态博客来说，这种方案比较适合长期扩展。文字页面仍然由 VuePress 管理，独立视觉页面交给 Three.js 实现，音乐数据通过 APlayer 和 Meting API 拉取。后续无论是替换模型、增加交互，还是加入新的可视化内容，都可以在这个结构上继续迭代。
