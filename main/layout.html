RecXX：基于 Reactor 模型的 C++ 高并发服务器

RecXX 是一个基于 C++17 开发的 Linux 高并发服务器项目。

项目采用非阻塞 Socket、epoll、Reactor 事件驱动模型和线程池，实现了 TCP 连接管理、HTTP 请求解析、静态资源响应、定时器管理、异步日志和优雅关闭等功能。

开发该项目的主要目的，是从底层理解一个高并发服务器如何完成连接建立、事件监听、数据收发、任务调度和资源回收，而不是简单调用现成的 Web 框架。

1. 项目简介

传统的阻塞式服务器通常为每个客户端连接创建一个线程。

当并发连接数量增加时，大量线程会带来明显的上下文切换、内存占用和调度开销。RecXX 使用事件驱动模型，由少量线程管理大量连接，从而提高服务器在高并发场景下的资源利用率。

项目主要技术栈如下：

客户端请求
    │
    ▼
监听 Socket
    │
    ▼
Acceptor 接收新连接
    │
    ▼
EventLoop 监听事件

    ├── 可写事件

主线程负责监听端口和接收新连接，工作线程负责处理已建立连接上的网络事件和业务任务。

通过将连接管理、I/O 处理和业务逻辑进行拆分，可以减少模块之间的耦合，并提高服务器的可扩展性。

3. Reactor 事件驱动模型

RecXX 使用 Reactor 模型处理网络事件。

Reactor 的核心思想是：服务器不主动阻塞等待某一个连接，而是统一监听多个连接。当某个连接变为可读、可写或发生异常时，再执行对应的处理逻辑。

事件循环的基本流程如下：

while (!quit_) {
    int eventCount = epoller.wait(events);

    for (int i = 0; i < eventCount; ++i) {
        Channel* channel = events[i].channel;

        if (channel->isReadable()) {
            channel->handleRead();
        }

        if (channel->isWritable()) {
            channel->handleWrite();
        }

        if (channel->hasError()) {
            channel->handleError();
        }
    }

    runPendingTasks();
}

每一个连接都通过 Channel 描述其关注的事件，并绑定对应的回调函数。

EventLoop 负责等待 epoll 事件，获取就绪连接，并分发到相应的处理函数。

这种设计避免了业务代码直接操作底层 epoll，使网络层逻辑更加清晰。

4. epoll 与非阻塞 I/O

服务器底层使用 epoll 监听 Socket 事件。

所有客户端连接都被设置为非阻塞模式：

int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

非阻塞模式下，单次 read 或 write 不保证完成全部数据处理，因此服务器需要循环读取或记录未发送的数据。

读取数据时：

while (true) {
    ssize_t n = read(fd, buffer, sizeof(buffer));

    if (n > 0) {
        inputBuffer.append(buffer, n);
    } else if (n == 0) {
        closeConnection();
        break;
    } else {
        if (errno == EAGAIN || errno == EWOULDBLOCK) {
            break;
        }

        handleError();
        break;
    }
}

写入数据时，如果发送缓冲区暂时写满，则保存剩余数据，并监听可写事件：

ssize_t n = write(fd, output.data(), output.size());

if (n < static_cast<ssize_t>(output.size())) {
    outputBuffer.append(output.data() + n, output.size() - n);
    channel.enableWriting();
}

当连接再次变为可写时，服务器继续发送剩余内容。

5. 多线程模型

RecXX 使用线程池提高多核 CPU 的利用率。

服务器启动后会创建多个工作线程，每个工作线程拥有独立的事件循环。

Main EventLoop
    │
    ├── Worker EventLoop 1
    ├── Worker EventLoop 2
    ├── Worker EventLoop 3
    └── Worker EventLoop 4

主线程接受新连接后，通过轮询方式将连接分配给不同的工作线程。

EventLoop* EventLoopThreadPool::getNextLoop() {
    EventLoop* loop = loops_[nextIndex_];

    nextIndex_ = (nextIndex_ + 1) % loops_.size();

    return loop;
}

每个连接从建立到关闭，通常由同一个工作线程负责处理。

这种 one loop per thread 的设计可以减少线程之间的数据竞争，也能避免多个线程同时操作同一个连接。

跨线程任务通过任务队列提交到目标事件循环：

void EventLoop::queueInLoop(Task task) {
    {
        std::lock_guard<std::mutex> lock(mutex_);
        pendingTasks_.push_back(std::move(task));
    }

    wakeup();
}

当其他线程提交任务后，通过 eventfd 唤醒正在等待 epoll 的事件循环。

6. HTTP 请求解析

服务器实现了基本的 HTTP/1.1 请求解析。

请求主要包含：

请求行
请求头
空行
请求体

例如：

GET /index.html HTTP/1.1
Host: localhost
Connection: keep-alive

解析器使用状态机处理请求：

enum class ParseState {
    RequestLine,
    Headers,
    Body,
    Finished,
    Error
};

请求行解析内容包括：

请求方法
请求路径
HTTP 版本

请求头使用键值结构保存：

std::unordered_map<std::string, std::string> headers;

服务器根据请求路径查找静态文件，并生成 HTTP 响应：

HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1024
Connection: keep-alive

对于不存在的资源，则返回：

HTTP/1.1 404 Not Found

7. Buffer 缓冲区设计

由于 TCP 是字节流协议，一次读取的数据可能不是完整的 HTTP 请求。

一个请求可能被拆分成多次到达，也可能一次读取到多个请求。因此项目实现了独立的 Buffer 类管理网络数据。

Buffer 主要维护三个区域：

已读取区域
可读取区域
可写入区域

常用操作包括：

buffer.readableBytes();
buffer.writableBytes();
buffer.append(data, length);
buffer.retrieve(length);
buffer.retrieveAll();

当尾部空间不足时，Buffer 会先尝试移动未处理的数据；如果仍然不足，再扩展底层存储空间。

这种方式减少了频繁的内存分配和数据复制。

8. 定时器与空闲连接清理

客户端可能建立连接后长时间不发送数据。

如果服务器一直保留这些连接，会持续占用文件描述符和内存资源。

RecXX 使用定时器记录连接的最后活跃时间，并定期清理超时连接。

每次连接发生读写事件时，都会刷新定时器：

timerManager.refresh(
    connectionId,
    std::chrono::seconds(30)
);

连接超过指定时间没有活动时，服务器会执行关闭操作。

定时器能够避免恶意连接或失效连接长期占用服务器资源。

9. 异步日志系统

网络线程不应因为日志写入而长时间阻塞。

RecXX 将日志消息先写入内存队列，再由独立线程批量写入文件。

业务线程
    │
    ▼
日志缓冲队列
    │
    ▼
日志线程
    │
    ▼
日志文件

日志内容包括：

时间
日志级别
线程编号
客户端地址
请求方法
请求路径
响应状态
错误信息

示例：

[2025-08-10 18:32:11] [INFO] server started at 0.0.0.0:8080
[2025-08-10 18:32:18] [INFO] new connection from 127.0.0.1
[2025-08-10 18:32:18] [INFO] GET /index.html 200

异步日志可以降低文件 I/O 对网络事件循环的影响。

10. 连接生命周期管理

一个连接通常经历以下过程：

创建连接
    │
    ▼
注册 epoll 事件
    │
    ▼
读取请求
    │
    ▼
解析请求
    │
    ▼
生成响应
    │
    ▼
发送数据
    │
    ├── 保持连接
    └── 关闭连接

连接关闭时，需要依次完成：

从 epoll 中移除事件
关闭文件描述符
删除定时器
释放输入输出缓冲区
释放连接对象

为了防止对象已经销毁但回调仍然执行，项目使用智能指针管理连接生命周期。

std::shared_ptr<TcpConnection>
std::weak_ptr<TcpConnection>

回调触发前先通过 weak_ptr 判断连接对象是否仍然有效，从而降低悬空指针和重复释放的风险。

12. 项目中的关键问题

12.1 TCP 粘包和拆包

TCP 不保留消息边界，因此不能假设一次 read 就能得到完整请求。

解决方式是把数据持续写入 Buffer，并根据 HTTP 协议格式判断请求是否完整。

12.2 非阻塞写入不完整

调用 write 时，数据不一定一次发送完成。

项目会保存剩余数据，并开启 EPOLLOUT 事件。数据全部发送完成后，再关闭可写事件监听，避免 epoll 持续触发。

12.3 跨线程任务调度

一个连接只能由所属事件循环线程操作。

其他线程需要修改连接状态时，不能直接调用，而是将任务加入目标事件循环的任务队列，再通过 eventfd 唤醒该线程。

12.4 连接重复关闭

读事件、写事件、错误事件和超时事件都可能触发关闭逻辑。

项目通过连接状态标记保证关闭流程只执行一次，避免重复关闭文件描述符或重复释放资源。

12.5 SIGPIPE 信号

客户端提前断开连接后，服务器继续写入可能触发 SIGPIPE，导致整个进程退出。

服务器启动时忽略该信号：

signal(SIGPIPE, SIG_IGN);

13. 项目优化

项目主要进行了以下优化：

  • 使用 epoll 管理大量网络连接；
  • 所有 Socket 使用非阻塞模式；
  • 使用多事件循环充分利用多核 CPU；
  • 使用 Buffer 减少频繁内存分配；
  • 仅在存在待发送数据时监听可写事件；
  • 使用 Keep-Alive 减少重复建立连接的开销；
  • 使用定时器清理长时间不活跃的连接；
  • 使用异步日志减少磁盘 I/O 阻塞；
  • 使用智能指针管理连接生命周期；
  • 将网络层和业务层解耦，方便扩展协议和业务逻辑。

16. 项目收获

通过 RecXX，我对 Linux 高并发网络编程有了更完整的理解。

相比只会使用 Web 框架，这个项目让我真正理解了一个请求从建立 TCP 连接，到进入 epoll，再到 HTTP 解析和响应发送的完整过程。

项目中涉及的知识包括：

  • Linux Socket 编程；
  • epoll 事件通知机制；
  • Reactor 网络模型；
  • 非阻塞 I/O；
  • 多线程和线程同步；
  • TCP 数据流处理；
  • HTTP 协议解析；
  • 定时器和连接管理；
  • 异步日志；
  • C++ 智能指针和对象生命周期；
  • CMake 项目构建。

这个项目并不是一个完整的生产级 Web 服务器，但它实现了高并发服务器最核心的基础组件，也为后续学习 RPC、分布式系统、消息队列和服务端开发建立了基础。

面试时可以这样介绍该项目：

RecXX 是我使用 C++17 开发的 Linux 高并发服务器。项目底层使用非阻塞 Socket 和 epoll，整体采用 Reactor 事件驱动模型。主线程负责接收连接，工作线程分别维护独立的 EventLoop，并通过轮询方式分配连接。服务器实现了 HTTP 请求解析、静态资源响应、Buffer 缓冲区、空闲连接定时清理和异步日志。开发过程中，我重点处理了非阻塞读写、TCP 粘包拆包、跨线程任务调度、连接生命周期以及重复关闭等问题。

结语

RecXX 是一次从底层出发的服务端开发实践。

它让我不再只把服务器理解为“接收请求并返回数据”，而是开始关注事件调度、线程模型、连接状态、缓冲区、资源回收和异常处理。

对于 C++ 服务端开发而言，真正重要的不只是实现一个能够运行的服务器，更是理解每一个连接在系统内部如何被创建、处理和安全地销毁。
