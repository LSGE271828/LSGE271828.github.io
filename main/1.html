ForgeAgent：基于 MCP 与 RAG 的代码仓库智能分析系统

ForgeAgent 是一个面向真实代码仓库的 AI Agent 系统。

它能够读取项目结构、检索代码、分析模块依赖、定位潜在问题、生成修改方案，并在受限环境中执行测试和验证任务。

与普通的代码问答系统不同，ForgeAgent 并不是一次性把整个仓库交给大模型，而是让 Agent 根据任务目标，主动选择检索、文件读取、符号搜索、命令执行和测试等工具，逐步完成分析过程。

项目主要关注以下问题：

大模型如何理解一个中大型代码仓库？
Agent 如何选择正确的工具？
长任务执行过程中如何保存状态？
工具调用失败后如何恢复？
如何限制 Agent 的执行权限？
如何判断任务是否真正完成？

1. 项目背景

大语言模型可以快速生成代码，但在真实工程环境中，仅靠一次提示词通常很难完成复杂任务。

一个代码仓库可能包含：

数百个源文件
复杂的目录结构
跨模块依赖关系
多个构建与测试命令
不同版本的配置文件
隐含在代码中的业务约束

如果直接将大量代码放入上下文，不仅会消耗大量 Token，还会使真正重要的信息被无关内容淹没。

因此，ForgeAgent 采用“按需检索、逐步执行、持续验证”的方式处理代码任务。

任务输入
 ↓
分析任务目标
 ↓
检索仓库结构
 ↓
定位相关模块
 ↓
读取关键代码
 ↓
生成执行计划
 ↓
调用工具完成任务
 ↓
验证结果

这种方式更接近开发者实际排查问题的过程。

2. 系统能力

ForgeAgent 当前主要支持以下类型的任务：

任务类型说明
仓库概览分析项目结构、技术栈和核心模块
代码检索根据自然语言定位相关文件与函数
依赖分析分析模块、类和函数之间的调用关系
错误定位结合日志和代码分析异常来源
修改建议生成结构化的代码修改方案
文档生成根据代码自动整理模块说明
测试执行在受限环境中运行指定测试命令
结果审查检查修改是否满足任务要求

例如，用户可以输入：

分析这个仓库的网络模块，并说明连接关闭时的资源释放流程。

Agent 会依次执行：

1. 获取项目目录结构
2. 搜索与 Connection、Socket、Channel 相关的文件
3. 读取连接对象的创建和销毁逻辑
4. 分析文件描述符、事件监听和定时器的释放顺序
5. 生成最终说明

对于更加复杂的任务：

定位登录接口偶发返回 401 的原因，并给出修改建议。

系统可能执行：

搜索认证相关文件
→ 查找 401 响应生成位置
→ 分析 Token 校验流程
→ 读取配置和中间件
→ 检查异常日志
→ 输出可能原因
→ 生成修改方案
→ 运行相关测试

4. Agent 任务循环

ForgeAgent 的核心是一个带有明确终止条件的任务循环。

while not state.finished:
  plan = planner.next_step(state)

  if plan.action == "tool":
      result = tool_executor.execute(
          name=plan.tool_name,
          arguments=plan.arguments,
      )

      state.add_observation(result)

  elif plan.action == "answer":
      review = reviewer.check(
          goal=state.goal,
          evidence=state.observations,
          answer=plan.content,
      )

      if review.passed:
          state.finished = True
          state.final_answer = plan.content
      else:
          state.add_feedback(review.feedback)

  if state.step_count >= state.max_steps:
      state.finished = True
      state.final_answer = build_partial_result(state)

每次循环主要完成四件事：

观察当前状态
决定下一步动作
调用工具或生成结果
检查任务是否已经完成

为了避免 Agent 无限循环，系统设置了：

最大执行步骤
最大工具调用次数
最大 Token 消耗
单个工具超时时间
任务总超时时间

5. Planner：任务规划

Planner 负责将复杂任务拆解为可以执行的步骤。

例如，面对任务：

分析服务器项目中可能存在的内存泄漏。

Planner 不会立即生成结论，而是先输出结构化计划：

{
"goal": "分析服务器项目中可能存在的内存泄漏",
"steps": [
  {
    "id": 1,
    "action": "scan_repository",
    "purpose": "获取项目结构和主要模块"
  },
  {
    "id": 2,
    "action": "search_code",
    "purpose": "查找动态内存和资源管理相关代码"
  },
  {
    "id": 3,
    "action": "inspect_lifecycle",
    "purpose": "检查连接和任务对象的生命周期"
  },
  {
    "id": 4,
    "action": "review_cleanup",
    "purpose": "检查异常分支中的资源释放逻辑"
  }
]
}

Planner 只负责决定“下一步应该做什么”，不会直接执行底层操作。

这样可以降低规划逻辑与工具实现之间的耦合。

6. Executor：工具执行

Executor 根据 Planner 的输出选择对应工具。

工具统一使用结构化输入和输出：

class ToolRequest(BaseModel):
  name: str
  arguments: dict


class ToolResult(BaseModel):
  success: bool
  content: str
  error: str | None = None
  metadata: dict = {}

例如代码搜索工具：

class SearchCodeInput(BaseModel):
  query: str
  file_pattern: str | None = None
  max_results: int = 20

工具返回：

{
"success": true,
"content": [
  {
    "path": "src/net/TcpConnection.cpp",
    "line_start": 84,
    "line_end": 126,
    "score": 0.91
  },
  {
    "path": "src/net/EventLoop.cpp",
    "line_start": 42,
    "line_end": 78,
    "score": 0.82
  }
]
}

所有工具都尽量保持单一职责。

search_code：搜索代码
read_file：读取文件
list_directory：查看目录
find_symbol：定位符号
run_test：运行测试
git_diff：查看修改

没有设计一个可以完成所有操作的万能工具，因为工具职责越模糊，模型越容易选择错误。

7. 基于 MCP 的工具连接

ForgeAgent 使用 MCP 统一接入外部工具和数据资源。

整体结构如下：

ForgeAgent
 │
 ▼
MCP Client
 │
 ├── Repository MCP Server
 ├── Test Runner MCP Server
 ├── Git MCP Server
 └── Documentation MCP Server

Repository MCP Server 主要提供：

读取目录
读取文件
搜索文本
查找符号
获取文件元信息

工具定义示例：

@mcp.tool()
def search_repository(
  query: str,
  file_pattern: str = "**/*",
  max_results: int = 20,
) -> list[dict]:
  """
  在代码仓库中搜索与查询内容相关的文件和代码片段。
  """

  return repository.search(
      query=query,
      file_pattern=file_pattern,
      max_results=max_results,
  )

通过 MCP，Agent 不需要了解底层工具的具体实现。

它只需要根据工具描述、参数结构和返回结果完成调用。

8. 代码知识库构建

为了让 Agent 能够处理规模较大的仓库，ForgeAgent 构建了代码知识库。

处理流程如下：

扫描仓库
 ↓
识别文件类型
 ↓
解析代码结构
 ↓
按函数、类和模块切分
 ↓
生成文本描述
 ↓
计算向量
 ↓
写入向量数据库

与普通文档不同，代码不能只按照固定字符数切分。

ForgeAgent 优先按照以下结构进行切分：

类
函数
方法
接口
配置段
模块说明

一个代码块会保存：

{
"path": "src/agent/planner.py",
"language": "python",
"symbol": "Planner.next_step",
"start_line": 35,
"end_line": 92,
"content": "...",
"summary": "根据当前任务状态生成下一步执行计划"
}

这样在检索时，系统不仅能返回代码，还能返回符号、文件和行号等信息。

9. 混合检索

单纯依赖向量检索，在处理函数名、异常码和配置项时效果并不稳定。

因此，ForgeAgent 使用混合检索：

向量语义检索
+
关键词检索
+
符号检索
+
结果重排序

例如搜索：

连接超时后在哪里关闭 Socket？

系统会同时检索：

connection timeout
close socket
idle timer
handleClose
removeConnection

最终结果根据以下信息进行排序：

语义相似度
关键词匹配程度
符号名称匹配
文件路径权重
调用关系相关性

对于明确的类名和函数名，关键词与符号检索优先级更高；对于自然语言描述的问题，语义检索占据更大权重。

10. 上下文管理

Agent 不会把所有检索结果一次性放入模型上下文。

系统将上下文分为四类：

系统规则
当前任务
执行历史摘要
本轮工具结果

任务执行过程中，旧的工具结果会被压缩为摘要：

{
"step": 4,
"summary": "TcpConnection::handleClose 负责触发关闭回调，
真正移除 Channel 的逻辑位于 TcpServer::removeConnectionInLoop。",
"evidence": [
  "src/net/TcpConnection.cpp:118-137",
  "src/net/TcpServer.cpp:91-112"
]
}

这样可以减少上下文长度，同时保留关键证据。

对于重要代码片段，系统会保存原始文件路径和行号，以便后续重新读取。

11. 状态与记忆

每个任务都有独立状态对象：

class AgentState(BaseModel):
  task_id: str
  goal: str

  current_step: int = 0
  max_steps: int = 20

  plan: list[dict] = []
  observations: list[dict] = []
  summaries: list[str] = []

  used_tools: list[str] = []
  generated_files: list[str] = []

  finished: bool = False
  final_answer: str | None = None

任务状态会在每次工具执行后保存。

即使服务中断，也可以根据状态继续执行，而不需要完全重新开始。

短期记忆用于保存当前任务信息，长期记忆则主要保存：

仓库结构摘要
稳定的项目约定
常用构建命令
历史任务结果
用户确认过的偏好

系统不会无限保存所有对话，而是只保留对后续任务有价值的信息。

12. Reviewer：结果审查

Planner 和 Executor 负责完成任务，Reviewer 负责判断结果是否可信。

Reviewer 会检查：

是否回答了原始问题
是否调用了必要工具
是否提供了代码证据
是否存在未经验证的推断
是否遗漏重要模块
是否满足输出格式

审查结果采用结构化形式：

{
"passed": false,
"score": 0.68,
"problems": [
  "缺少连接关闭异常分支的分析",
  "没有验证定时器是否同步删除"
],
"next_action": "继续读取 TimerManager 和异常处理代码"
}

如果 Reviewer 判断证据不足，任务会重新进入执行循环。

这可以减少 Agent 在信息不完整时过早生成结论。

13. 工具失败与任务恢复

工具调用并不一定总是成功。

常见失败包括：

文件不存在
搜索结果为空
命令执行超时
测试环境缺少依赖
工具参数格式错误
仓库权限不足

工具失败时，Executor 会返回标准错误：

{
"success": false,
"error_type": "FILE_NOT_FOUND",
"error": "src/server/config.py does not exist",
"recoverable": true
}

Planner 根据错误类型决定下一步：

文件不存在
→ 重新搜索相似路径

测试失败
→ 读取测试日志

参数错误
→ 修正参数后重试

权限不足
→ 请求人工确认或终止任务

系统会限制同一个工具以相同参数重复调用，避免无意义循环。

14. 安全与权限控制

代码 Agent 不仅能够读取信息，还可能执行命令或修改文件，因此必须限制执行边界。

ForgeAgent 将工具分为三个等级：

等级工具执行方式
低风险搜索、读取、目录浏览自动执行
中风险创建文件、修改代码、运行测试受限执行
高风险删除文件、推送代码、执行系统命令人工确认

Shell 工具采用命令白名单：

ALLOWED_COMMANDS = {
  "pytest",
  "npm test",
  "npm run lint",
  "cmake --build build",
  "ctest",
}

以下命令默认禁止：

rm -rf
sudo
shutdown
curl | sh
任意网络下载并执行
访问项目目录之外的文件

代码执行在隔离目录中进行，并设置：

CPU 时间限制
内存限制
执行超时
文件系统范围
网络访问限制

15. 可观测性

每个 Agent 任务都会生成完整执行轨迹。

{
"task_id": "agent-20260711-001",
"steps": [
  {
    "step": 1,
    "agent": "planner",
    "action": "search_code",
    "duration_ms": 125
  },
  {
    "step": 2,
    "agent": "executor",
    "tool": "read_file",
    "duration_ms": 38
  },
  {
    "step": 3,
    "agent": "reviewer",
    "result": "continue",
    "duration_ms": 412
  }
]
}

系统主要记录：

任务总耗时
模型调用次数
工具调用次数
工具失败次数
每一步输入和输出
Token 消耗
任务终止原因
人工确认记录

这些信息既可以用于调试，也可以用于分析 Agent 经常失败的任务类型。

16. Agent 评测

为了避免只依靠人工观察判断效果，ForgeAgent 将任务整理为可重复执行的评测样例。

一个评测任务包括：

{
"id": "repo-analysis-01",
"question": "说明项目中线程池的任务调度流程",
"required_evidence": [
  "ThreadPool::submit",
  "Worker::run"
],
"expected_tools": [
  "search_repository",
  "read_file"
],
"forbidden_actions": [
  "write_file",
  "run_shell"
]
}

主要评测指标包括：

指标含义
任务完成率是否完成用户目标
工具选择准确率是否选择正确工具
参数正确率工具参数是否有效
证据覆盖率是否找到关键代码
恢复成功率工具失败后能否恢复
平均执行步骤完成任务所需步骤数
无效调用率是否存在重复或无关调用
安全违规率是否触发禁止操作

评测结果不仅用于比较不同模型，也用于判断提示词、检索策略和工具描述的修改是否真正有效。

}

20. 一次完整执行过程

下面是一个典型任务的执行过程。

任务：

Planner 判断需要定位：

第三步：读取代码

Agent 分别读取相关函数，并记录路径和行号。

21. 设计取舍

为什么不直接把整个仓库交给模型

大型仓库往往超过模型上下文限制，即使能够放入，也会造成较高成本和信息干扰。

按需检索更符合真实代码分析流程。

为什么使用 Planner、Executor 和 Reviewer

单个 Agent 同时负责规划、执行和判断结果，容易过早得出结论。

职责拆分后，可以让不同阶段拥有更明确的约束。

为什么工具输出必须结构化

纯文本工具结果难以判断错误类型，也不方便后续自动评测。

结构化输出能够减少解析歧义。

为什么不让 Agent 自动执行所有命令

代码 Agent 具有真实操作能力。

执行范围越大，越需要权限控制、沙箱和人工确认。

为什么保留完整执行轨迹

没有执行轨迹，就很难判断 Agent 究竟在哪一步出现问题。

对于长任务而言，可观测性和模型能力同样重要。

22. 项目总结

ForgeAgent 的重点并不是让大模型生成更多代码，而是让模型在受控环境中完成完整任务。

一个可以实际运行的 Agent 系统，需要同时处理：

任务规划
工具调用
知识检索
上下文管理
状态保存
失败恢复
权限控制
执行追踪
结果评测

在实现过程中，我逐渐认识到，Agent 的效果并不只取决于模型本身。

很多失败来自工具设计不清晰、检索结果不准确、状态没有保存、终止条件模糊，或者系统无法判断任务是否真正完成。

因此，ForgeAgent 更像是一套围绕大语言模型构建的任务执行系统。

模型负责理解和决策，工具负责与外部环境交互，状态模块负责保存过程，Reviewer 负责验证结果，而安全模块则限制整个系统能够采取的行动。

这也是我对 AI Agent 工程的基本理解：

Agent 不只是会调用工具的大模型，而是一套能够规划、执行、观察、恢复和验证的完整系统。施胸外按压的获益远大于潜在风险。
